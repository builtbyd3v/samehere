# v1.5 — Direct messaging + collab posts (2026-07-03)

Spec for the two interaction gaps called out in the UI batch: **DMs** (no private 1:1 channel today) and **group/collab posting** (no shared authorship). Ship DMs first; collab posts build on the same trust layer (blocks, reports, notifications).

---

## Why this order

1. **DMs** unblock the biggest daily friction: follow + public comment is not enough for coordination, intros, or sensitive topics.
2. **Collab posts** need identity + notification plumbing DMs already force us to build.
3. Both reuse **blocks** (either direction hides thread + rejects new messages), **reports**, and **in-app notifications** (no realtime in v1.5 either — poll on load, same as today).

---

## 1. Direct messaging (v1.5 flagship)

### Scope (MVP)

| In | Out |
|----|-----|
| 1:1 text threads | Group chats |
| Thread list + chat view | Read receipts, typing indicators |
| New message from profile or post author row | Media / links preview cards |
| Block-aware (hide thread, reject sends) | Email push |
| Report message (→ `reports` pattern, new `message_id` nullable on reports or `message_reports` table) | Realtime (Supabase Realtime deferred) |
| `.edu` users only (inherits auth gate) | Message requests from non-mutuals (phase 2 — see below) |

### Schema (draft)

```sql
create table conversations (
  id uuid primary key default gen_random_uuid(),
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now() -- bump on new message for sort
);

-- Exactly two members per conversation in MVP; unique pair via ordered user ids.
create table conversation_members (
  conversation_id uuid not null references conversations(id) on delete cascade,
  user_id uuid not null references profiles(id) on delete cascade,
  last_read_at timestamptz,
  created_at timestamptz not null default now(),
  primary key (conversation_id, user_id)
);

create unique index conversation_pair_unique on conversation_members (
  least(user_id, ...), greatest(user_id, ...)  -- implement via definer get_or_create_dm(a,b)
);

create table messages (
  id uuid primary key default gen_random_uuid(),
  conversation_id uuid not null references conversations(id) on delete cascade,
  sender_id uuid not null references profiles(id) on delete cascade,
  content text not null check (char_length(content) between 1 and 2000),
  created_at timestamptz not null default now()
);

create index messages_conversation_created on messages(conversation_id, created_at desc);
```

**RLS sketch**

- `conversation_members` select: `auth.uid() = user_id`.
- `messages` select: member of parent conversation AND no block between viewer and sender (or check in definer list function).
- `messages` insert: `auth.uid() = sender_id` AND member AND not blocked.
- No anon access.

**API / routes**

- `GET /messages` — thread list (preview + unread count from `last_read_at`).
- `GET /messages/[id]` — paginated messages, cursor on `created_at`.
- `POST /messages` — `get_or_create_conversation(recipient_id)` + insert message + notification trigger.
- `PATCH /messages/[id]/read` — bump `last_read_at`.
- Entry points: **Message** button on profile (hidden if blocked), optional on post card overflow menu.

### Phase 2 (post-MVP)

- **Message requests** for non-mutuals (like Instagram): lands in Requests inbox; accept opens thread.
- Unread badge on navbar (Phase 14 notifications bell already planned — wire DM count there).

### Abuse / safety

- Coarse rate limit: max N new conversations / hour per user (definer or trigger).
- Report message → triage in Supabase dashboard (same as posts).
- Block removes thread from list both ways; existing messages not delivered to blocker.

### UI surfaces (coming soon → ship)

- `/messages` — inbox + thread view (mobile-first).
- Nav: envelope icon next to notifications bell.
- Settings → Messaging: “Allow messages from” (everyone / followers only) — can default everyone for MVP.

## 2. Collab posts (v1.5 follow-on)

### Scope (MVP)

| In | Out |
|----|-----|
| One co-author per post (invite at compose time) | 3+ authors |
| Both avatars on post card | Split heatmap points negotiation beyond “both get post credit” |
| Co-author must accept invite before post goes live | Open collaboration without accept |
| Notification on invite + accept | Edit after publish |

### Schema (draft)

```sql
create table post_collaborators (
  post_id uuid not null references posts(id) on delete cascade,
  user_id uuid not null references profiles(id) on delete cascade,
  role text not null check (role in ('coauthor')),
  status text not null check (status in ('pending','accepted','declined')),
  invited_by uuid not null references profiles(id),
  created_at timestamptz not null default now(),
  primary key (post_id, user_id)
);
```

**Flow**

1. Composer: “Add co-author” → search @username → creates **draft** post + pending row.
2. Invitee gets notification; accept → post publishes (or was held in `draft` until accept).
3. Feed shows stacked avatars + “Priya & Dev” attribution.
4. `log_contribution` for `post` action: both author and accepted co-author get point if 150+ chars (once per user per day — existing dedupe).

**RLS**

- Insert collaborator: only post owner, not blocked either way.
- Accept/decline: only invitee.
- Post select: unchanged visibility on primary author; co-author name visible on accepted posts only.

### Why after DMs

Collab invites are awkward without a fallback channel (“accept my collab invite”). DMs + notifications make the invite flow tolerable.

## 4. Acceptance criteria (v1.5)

**DMs**

- [ ] User A messages User B; B sees thread in inbox within one page load.
- [ ] Block either way: thread hidden, new sends rejected.
- [ ] Report message stored for triage.
- [ ] No service_role in app routes.

**Collab**

- [ ] Invite → accept → post visible on both profiles’ post lists.
- [ ] Decline → post never publishes (or author-only draft remains deletable).
- [ ] Private account rules: co-author must follow or be followed? **Decision:** invite only if mutual follow OR either direction accepted follow — avoids amplifying private content to strangers.

---

## 5. Open decisions (for product review)

1. DMs: mutual follow required at MVP, or open to any authenticated user?
2. Collab: max pending invites per user per day?
3. Heatmap: co-author both earn full 5 pts or split 3+2?

Default recommendations: **DMs open to any authed user** (block is the escape hatch); **3 pending collab invites/day**; **both earn full post point** (rare enough, rewards real collaboration).
